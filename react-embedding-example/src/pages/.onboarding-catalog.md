Thin page-composition component that wires the onboarding guides catalog UI to its API endpoints with zero custom logic.

## Key Components

- **`OnboardingCatalogPage`** — Default page export; renders the onboarding catalog by composing two library components and injecting endpoint config.
- **`DevSectionPage`** — Provides page chrome (hero banner + back button) scoped to the `"onboarding"` section key.
- **`OnboardingGuidesCatalogView`** — Library component that owns all rendering logic: RAG search, section pills, guide grid, and internal data fetching (reads `?section=` from the URL).
- **`EP`** — Centralized endpoint config; supplies `EP.onboarding` and `EP.onboardingSections` as the only page-level concern.

## Usage Example

```typescript
// Route registration (e.g. React Router / Next.js)
import { OnboardingCatalogPage } from './pages/onboarding-catalog'

// <Route path="/onboarding-guides" element={<OnboardingCatalogPage />} />

// The component itself needs no props — all config
// flows in via EP and the URL's ?section= query param.
export function OnboardingCatalogPage() {
  return (
    <DevSectionPage sectionKey="onboarding">
      <OnboardingGuidesCatalogView
        guidesEndpoint={EP.onboarding}
        sectionsEndpoint={EP.onboardingSections}
        basePath="/onboarding-guides"
      />
    </DevSectionPage>
  )
}
```

## Notes

- **Config-only pattern** — follows the same shape as roadmap, releases, and delivery pages; no local state, no data-fetching, no business logic lives here.
- To point the catalog at different API routes, update `EP.onboarding` / `EP.onboardingSections` in the shared endpoint config rather than modifying this file.
