Vite alias stub for `next/font/google` that prevents build crashes when Next.js font compiler macros are transitively imported in a non-Next.js bundler context.

## Key Components

- **`FontResult`** — Type describing the shape returned by Next.js font functions (`className`, `variable`, `style.fontFamily`)
- **`noop`** — Zero-argument factory returning an empty `FontResult`, used as a safe replacement for all font constructors
- **`Azeret_Mono`** / **`DM_Sans`** — Named exports matching the specific fonts used in the library's `./fonts` entry
- **`default` (Proxy)** — Catch-all default export that maps any property access to `noop`, covering any other Google font name

## Usage Example

This file is not imported directly — it is wired up as a Vite path alias in `vite.config.ts`:

```typescript
// vite.config.ts
resolve: {
  alias: {
    'next/font/google': '/src/stubs/next-font-google.ts',
  },
}
```

Any code that calls a Google font function will silently receive an empty result:

```typescript
import { DM_Sans } from 'next/font/google'

const font = DM_Sans({ subsets: ['latin'] })
// font.className  → ''
// font.variable   → ''
// font.style.fontFamily → ''
```

> **Note:** Fonts in this project are applied via a CSS `./styles` import, not via JS font constructors. This stub is a **defensive guard only** — it ensures transitive imports don't crash the Vite build, not that font styles are applied at runtime.
