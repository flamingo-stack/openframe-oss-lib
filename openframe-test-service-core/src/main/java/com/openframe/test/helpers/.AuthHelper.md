Provides thread-safe management of authentication cookies for test flows, using `ThreadLocal` storage to isolate session state per thread.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `cookies` | `ThreadLocal<Map<String, String>>` | Per-thread cookie storage ensuring test isolation |
| `getCookies()` | `static` method | Returns current thread's cookies, lazily initializing via `AuthFlow.login()` if absent |
| `setCookies(Map)` | `static` method | Manually overrides the current thread's cookie map |
| `clearCookies()` | `static` method | Removes cookies from the current thread (important for cleanup) |

## Usage Example

```java
// Automatically logs in and retrieves cookies for the current thread
Map<String, String> sessionCookies = AuthHelper.getCookies();

// Inject cookies into an API request
given()
    .cookies(AuthHelper.getCookies())
    .when()
    .get("/api/resource")
    .then()
    .statusCode(200);

// Override cookies manually (e.g., for a different user session)
AuthHelper.setCookies(customCookies);

// Clean up after test to avoid state leakage between threads
AuthHelper.clearCookies();
```

## Notes

- **Lazy initialization**: `getCookies()` calls `AuthFlow.login(UserConfig.getUser())` only once per thread, reducing redundant login requests across tests.
- **Thread safety**: `ThreadLocal` ensures parallel test execution does not cause session bleed between threads.
- Always call `clearCookies()` in a `@AfterEach` or teardown method to prevent stale session data from affecting subsequent tests.
