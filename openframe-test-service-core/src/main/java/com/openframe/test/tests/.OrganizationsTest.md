JUnit 5 integration test suite validating the full CRUD lifecycle for the Organizations API in the OpenFrame platform.

## Key Components

| Method | Tag | Order | Description |
|--------|-----|-------|-------------|
| `testCreateOrganization()` | `create` | 1 | Creates an organization and validates all response fields including contact/mailing address matching |
| `testListOrganizations()` | `read` | 2 | Retrieves all organizations and asserts non-empty list with valid field presence |
| `testRetrieveOrganization()` | `read` | 3 | Fetches a single organization by `organizationId` and compares it against the list result |
| `testUpdateOrganization()` | `update` | 4 | Updates a non-default organization and verifies the response matches the update request |
| `testArchiveOrganization()` | `delete` | 5 | Archives an organization and confirms it no longer appears in the active list |

**Tags:** `@Tag("oss")`, `@Tag("saas")` — runs against both deployment targets.

## Usage Example

```java
// Run only create tests
@Tag("create")
mvn test -Dgroups="create"

// Run full Organizations suite ordered by lifecycle
@TestMethodOrder(MethodOrderer.OrderAnnotation.class)
// Tests execute: Create → List → Get → Update → Archive

// Example assertion pattern used internally
assertThat(organization)
    .usingRecursiveComparison()
    .ignoringFields("id", "organizationId", "isDefault", "createdAt", "updatedAt",
                    "status", "statusChangedAt", "contactInformation.mailingAddress")
    .isEqualTo(request);
```

## Notes

- Extends `BaseTest` for shared setup/teardown and API client configuration
- Uses `OrganizationGenerator` for deterministic test data generation
- Recursive comparison via AssertJ intentionally ignores server-generated fields (`id`, `createdAt`, `updatedAt`, etc.)
- Tests are **order-dependent** — each step relies on data created by the previous one
