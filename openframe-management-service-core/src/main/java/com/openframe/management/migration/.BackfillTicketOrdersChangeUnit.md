A Mongock database migration change unit that backfills LexoRank-based ordering values for existing tickets that lack an `order` field, grouped by ticket status.

## Key Components

| Component | Description |
|-----------|-------------|
| `@ChangeUnit(id = "backfill-ticket-orders", order = "002")` | Registers this as the second migration in the Mongock changelog |
| `execution()` | Entry point — iterates all `TicketStatus` values and triggers backfill per status column |
| `backfillColumn()` | Queries tickets with a null `order` field for a given status/tenant, assigns sequential LexoRank values sorted by `createdAt DESC` |
| `assignOrder()` | Applies the computed LexoRank string to a single ticket document via `mongoTemplate.updateFirst()` |
| `rollback()` | No-op rollback — intentionally left empty as rank assignment is non-destructive |

## Usage Example

This migration runs automatically via Mongock on application startup. No manual invocation is needed. The logic it encodes:

```java
// For each TicketStatus (e.g., OPEN, IN_PROGRESS, DONE)
// Find tickets missing an order field, sorted newest-first
Query query = new Query(Criteria.where("tenantId").is(tenantId)
        .and("status").is(status)
        .and("order").is(null));
query.with(Sort.by(Sort.Direction.DESC, "createdAt"));

// Assign LexoRank values starting from the middle bucket
LexoRank rank = LexoRank.middle();
for (Ticket ticket : tickets) {
    assignOrder(mongoTemplate, ticket.getId(), rank.format());
    rank = rank.genNext(); // e.g., "0|hzzzzz:" → "0|i00007:"
}
```

> **Note:** LexoRank ordering enables drag-and-drop ticket reordering within Kanban columns without requiring full list re-indexing. This migration ensures all pre-existing tickets receive an initial rank before that feature is used.
