Defines canonical URL parameter keys for dev-center sections, serving as the single source of truth shared between the chrome registry (which **writes** query params) and list views (which **read** them) to prevent silent filtering mismatches.

## Key Components

**`DEV_SECTION_PARAM_KEYS`** — a `const` object of URL parameter key strings used across dev-center sections:

| Key | URL Param | Used By |
|-----|-----------|---------|
| `search` | `?search=` | All dev-center sections (free-text search) |
| `status` | `?status=` | Roadmap & Help Center ticket filters |
| `releaseStatus` | `?release_status=` | Product releases stability-tier filter |
| `deliveryTaskType` | `?task_type=` | Delivery (bug-fix / enhancement) task-type filter |

## Usage Example

```typescript
import { DEV_SECTION_PARAM_KEYS } from '@/utils/dev-section-param-keys'

// In the chrome registry — WRITING params to the URL
const params = new URLSearchParams()
params.set(DEV_SECTION_PARAM_KEYS.status, 'in-progress')
params.set(DEV_SECTION_PARAM_KEYS.search, query)

// In a list view — READING params from the URL
const status = searchParams.get(DEV_SECTION_PARAM_KEYS.status)
const search = searchParams.get(DEV_SECTION_PARAM_KEYS.search)
```

> **Why this matters:** Both the chrome registry (`OPENFRAME_DEV_SECTIONS`) and the list views (`RoadmapView`, `ProductReleasesView`, `DeliveryLists`) import from this file rather than re-declaring string literals. This makes key mismatches structurally impossible — if the written key and the read key ever diverge, filtering breaks silently with no runtime error.
