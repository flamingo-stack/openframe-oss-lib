A scroll-to-element utility that drives a cancellation-proof, frame-by-frame tween instead of native smooth scrolling — solving the browser scroll-anchoring bug that causes `scrollTo({behavior:'smooth'})` to jump after the first use.

## Key Components

- **`scrollElementIntoView(target, options)`** — Main export. Scrolls the page so the target element lands at the top of the viewport, accounting for sticky headers, layout shifts, and reduced-motion preferences.
- **`ScrollElementIntoViewOptions`** — Configuration interface with `headerOffset`, `behavior`, `adjustTargetY`, and `durationMs` fields.
- **`cancelActiveScroll()`** — Internal function that cancels any in-flight tween and removes gesture listeners. Only one scroll animation runs at a time.
- **`easeOutCubic(t)`** — Internal easing function applied per frame for natural deceleration.
- **`computeTarget()`** — Recomputes the pixel target on every frame via `getBoundingClientRect()`, tracking elements that are still settling (e.g., an expanding drawer).

## Usage Example

```typescript
import { scrollElementIntoView } from '@/utils/scroll-into-view'

// Basic scroll with sticky header offset
const el = document.getElementById('my-section')
scrollElementIntoView(el, { headerOffset: 96 })

// Instant jump (e.g., deep-link landing)
scrollElementIntoView(el, { behavior: 'instant', headerOffset: 96 })

// Compensate for a sibling layout shift mid-tween
scrollElementIntoView(el, {
  headerOffset: 96,
  durationMs: 400,
  adjustTargetY: (raw) => raw - 48, // drawer collapsing adds 48px above
})
```

> **Why not `scrollTo({ behavior: 'smooth' })?`** Browser scroll anchoring (triggered by DOM mutations like drawer expansion or image load) calls an internal "abort ongoing smooth scroll" step on every correction — making native smooth scroll jump on every repeat. This utility bypasses that entirely by issuing instant `scrollTo` writes each rAF frame, which anchoring cannot abort.
