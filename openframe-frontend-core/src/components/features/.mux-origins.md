Server-safe module exporting Mux CDN origin constants, intentionally isolated from client-side code to prevent Next.js client-reference poisoning during server-side builds.

## Key Components

| Export | Value | Purpose |
|--------|-------|---------|
| `MUX_STREAM_ORIGIN` | `https://stream.mux.com` | HLS playback, segments, and per-asset MP4 renditions |
| `MUX_IMAGE_ORIGIN` | `https://image.mux.com` | Server-generated thumbnails |

## Usage Example

```typescript
import { MUX_STREAM_ORIGIN, MUX_IMAGE_ORIGIN } from '@/lib/config/mux-origins'

// Safe to use in server-side code (API routes, webhook handlers, etc.)
const streamHostname = new URL(MUX_STREAM_ORIGIN).hostname
// → 'stream.mux.com'

const thumbnailUrl = `${MUX_IMAGE_ORIGIN}/${playbackId}/thumbnail.jpg`
const playbackUrl  = `${MUX_STREAM_ORIGIN}/${playbackId}.m3u8`
```

> **Why a separate module?** When these constants lived inside `use-video-warmup.ts` (a `'use client'` file), Next.js replaced them with client-function stubs at build time. Any server-side code calling `new URL(MUX_STREAM_ORIGIN)` would throw `TypeError: Invalid URL` at Vercel build. Isolating constants here — with no `'use client'` directive — preserves the server-safe import path while `use-video-warmup.ts` re-exports them for backward compatibility with client callers.
