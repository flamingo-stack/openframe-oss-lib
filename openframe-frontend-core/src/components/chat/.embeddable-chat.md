Portable, lib-safe port of the hub's `<GlobalAskAI>` chat panel component, decoupled from all hub-only imports by routing navigation, identity, and transport decisions through a `useRequiredChatRuntime()` context hook.

## Key Components

| Export | Type | Description |
|--------|------|-------------|
| `EmbeddableChat` | Component | Main embeddable chat panel with controlled/uncontrolled open state |
| `EmbeddableChatProps` | Interface | Full prop surface for the component |
| `dispatchSlashCommandAction` | Function | Maps slash-command action IDs (`browse`, `search`, `find`, `display`) to `ChatInputRef` mutations |
| `resolveChipIcon` | Function | Resolves a `ChatSource` to a Lucide icon via the icon registry |
| `SKELETON_ROW_VARIANTS` | Constant | Deterministic width palette for loading skeleton rows |
| `FALLBACK_TOP_RETRIEVED` | Constant | Number of fallback source chips shown when the model cites no sources (default: `3`) |

## Usage Example

```typescript
// Host wires the runtime once at root
<HubRuntimeProvider source="hub" user={currentUser}>
  <EmbeddableChat
    emptyStateGreeting="How can I help you today?"
    suggestedQueries={['How do I reset a password?', 'Show open tickets']}
    enabledRagTableIds={enabledTableIds}
    modes={{
      guide: { tableIdForDocumentType },
      mingo: { agentId: 'mingo-agent' },
    }}
    defaultActiveMode="guide"
    extras={{ buildProgramProps, buildProductReleaseProps }}
  />
</HubRuntimeProvider>

// Controlled open state
<EmbeddableChat
  open={isOpen}
  onOpenChange={setIsOpen}
  showInternalTrigger={false}
/>
```

## Runtime Substitutions

| Hub Original | Lib Replacement |
|---|---|
| `useAuth()` | `useRequiredChatRuntime().user` |
| `currentPlatform()` | `useRequiredChatRuntime().source` |
| `useNavLink` / `NavLinkAnchor` | `handleChatNavClick` + `NavLinkAnchorViaRuntime` |
| `useDocChat(source)` | `useSseChatAdapter()` (source from runtime) |
## Agent mode (OpenFrame AI agents — Fae / Mingo)

Reuse the SAME `<EmbeddableChat>` to render a global OpenFrame AI agent. No new
component, no agent-specific render path — an "agent mode" just OVERRIDES the
empty-state URL so the chat fetches the agent's display config (greeting +
suggested prompts + identity) from the public `/api/ai-agents/[slug]` endpoint,
which is byte-compatible with the `EmptyStateConfig` wire shape.

- Pass `activeAgentSlug` (e.g. `"fae"`) to `<EmbeddableChat>`. **Works in BOTH
  regular (host) and embedded modes** — there is no navigation-mode gate.

```tsx
<EmbeddableChat activeAgentSlug="fae" />
```

**Default routes, fully overridable.** Agent mode works with ZERO endpoint
wiring — the component falls back to a built-in default route. Resolution order
for the agent display-config route (highest → lowest precedence):

1. `aiAgentConfigUrl` prop on `<EmbeddableChat>` (per-mount override)
2. `runtime.endpoints.aiAgentConfigUrl(slug)` (shared runtime override)
3. built-in default `` (slug) => `/api/ai-agents/${encodeURIComponent(slug)}` ``

Same-origin embedders need nothing; cross-origin embedders behind a proxy set
`runtime.endpoints.aiAgentConfigUrl` (or the prop) to their absolute proxied
path. Every other chat route (`chatStreamUrl`, `commandsUrl`, …) comes from
`runtime.endpoints` and is likewise overridable.

DISPLAY-ONLY this phase: retrieval still resolves server-side from the platform
(the chat route derives `source` from the deployment, not the client) — the
agent customizes the greeting + suggested-prompt chips, not the corpus. In mingo
mode, `mingoWelcome` still owns its own welcome surface.
