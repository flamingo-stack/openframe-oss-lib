Renders inline entity cards within the Flamingo chat interface by dispatching `ChatRef` objects to their corresponding card components based on document type.

## Key Components

### Types & Interfaces

- **`ChatCardDispatchExtras`** — Host-supplied builders for hub-side derived props (`programConfigs`, `buildProductReleaseCardProps`, `buildOgPlaceholderUrl`)
- **`ChatCardRenderOptions`** — Per-render options threaded from the chat shell (`baseRoute`, `isNewTab`, `discuss`, `extras`)

### Internal Components

- **`GitHubChatCard`** — Renders GitHub activity cards (commits, PRs, reviews) using `MingoInfoCard` with review-state status pills
- **`githubKindIcon(kind)`** — Maps `GitHubActivityKind` to the appropriate icon component
- **`reviewStateToVariant(state)`** — Maps `PrReviewState` to a `MingoInfoCardStatus` variant string

### Public Entry Points

- **`renderChatInlineEntityCard(ref, opts)`** — Top-level dispatch; hoists video refs to block cards, otherwise routes to compact inline cards
- **`<ChatCardLoader />`** — Handles fetch, skeleton, and render lifecycle for fetch-mode card entries

## Usage Example

```typescript
// Rendering an inline entity card inside the chat shell
import { renderChatInlineEntityCard } from './dispatch'

const card = renderChatInlineEntityCard(chatRef, {
  isNewTab: computeIsNewTab(chatRef.url, runtime),
  extras: {
    programConfigs: { podcast: podcastConfig },
    buildProductReleaseCardProps: (release) => buildProductReleaseCardProps(release),
    buildOgPlaceholderUrl: (title) => buildOgPlaceholderUrl(title),
  },
})
```

> **Note:** Hub-only concerns (URL building, platform detection, table ID resolution) are intentionally excluded. Pre-composed URLs arrive via `ref.url` from the server's RAG mapper, and host-side extras are injected at the shell boundary via `ChatCardRenderOptions.extras`.
