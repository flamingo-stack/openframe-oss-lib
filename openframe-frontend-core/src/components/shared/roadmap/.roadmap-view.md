Self-contained roadmap list surface that fetches and renders roadmap items with filtering, SSR hydration, and live vote patching.

## Key Components

- **`RoadmapView`** — Primary export; orchestrates data fetching via `useSelfFetch`, reads URL search params for filtering, and delegates rendering to `RoadmapGrid`
- **`RoadmapViewProps`** — Configuration interface exposing API routes (`endpoint`, `buildRefreshUrl`, `votingOptions`), SSR hydration (`initialItems`), and URL param keys (`searchParamKey`, `statusParamKey`)

## Behavior Notes

- **Filtering**: Reads `search` and `status` from URL params (written by the section chrome `DevSectionView`) and folds them into the fetch URL — the URL acts as the cache key, triggering refetches on filter changes
- **SSR hydration**: `initialItems` is memoized into a stable `initialData` reference to prevent the `useSelfFetch` re-sync effect from clobbering optimistic vote patches
- **Loading states**: Shows `RoadmapGridSkeleton` only during the first fetch (no data yet); malformed responses without `items` render an empty grid rather than a stuck skeleton
- **Live vote patching**: `onItemUpdate` patches the updated task back into local state after a per-task refresh, keeping vote counts live without a full list reload

## Usage Example

```typescript
// Minimal — uses all defaults (/api/roadmap, standard param keys)
<RoadmapView />

// Full configuration with SSR hydration and custom API routes
<RoadmapView
  endpoint="/api/my-roadmap"
  initialItems={serverSideItems}
  showLeftMargin
  buildRefreshUrl={(id) => `/api/my-roadmap/${id}`}
  votingOptions={{ voteEndpoint: '/api/vote', storageKey: 'roadmap-votes' }}
  searchParamKey="q"
  statusParamKey="status"
/>
```
