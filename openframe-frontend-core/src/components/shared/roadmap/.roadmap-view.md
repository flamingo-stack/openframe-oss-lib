Self-contained roadmap list surface that fetches, filters, and renders roadmap items via URL-driven search/status parameters.

## Key Components

| Export | Type | Description |
|--------|------|-------------|
| `RoadmapView` | Component | Main container that owns data fetching and URL param sync |
| `RoadmapViewProps` | Interface | Configuration contract for API routes and SSR hydration |

**Internal dependencies:**
- `useSelfFetch` — handles GET requests, loading/error state, and optimistic patch via `setData`
- `RoadmapGrid` — pure controlled list renderer
- `RoadmapGridSkeleton` — shown only during initial (no-data) fetch
- `DEV_SECTION_PARAM_KEYS` — shared param registry ensuring the section chrome and this view read/write the same URL keys

## Key Behaviors

- **URL-as-cache-key**: `search` and `status` query params are folded into the fetch URL, so filtering triggers a refetch automatically
- **SSR hydration**: `initialItems` is memoized into `initialData` to prevent the `useSelfFetch` sync effect from clobbering optimistic vote patches
- **Optimistic vote patch**: `onItemUpdate` merges a refreshed single task back into the list without a full refetch
- **Skeleton guard**: Skeleton renders only while the first fetch is in flight; an empty `items` array still renders the grid

## Usage Example

```typescript
import { RoadmapView } from './roadmap-view'

// Minimal — uses all defaults (/api/roadmap, shared param keys)
<RoadmapView />

// Fully configured
<RoadmapView
  endpoint="/api/my-roadmap"
  initialItems={ssrItems}
  showLeftMargin
  buildRefreshUrl={(id) => `/api/my-roadmap/${id}`}
  votingOptions={{ voteEndpoint: '/api/vote', storageKey: 'roadmap-votes' }}
  searchParamKey="q"
  statusParamKey="stage"
/>
```

> **Note:** `searchParamKey` and `statusParamKey` **must** match the keys written by the section chrome (`DevSectionView`). Mismatches silently break filtering — the shared `DEV_SECTION_PARAM_KEYS` registry is the single source of truth for defaults.
