Renders the delivery section body with two filterable tables — "Recently Completed" and "Active Tasks" — fetching data from separate API endpoints and syncing filter state via URL search params.

## Key Components

- **`DeliveryLists`** — Main exported component that reads `search` and `task_type` URL params, fetches both delivery buckets in parallel, and renders two `<DeliveryTable>` instances
- **`DeliveryListsProps`** — Interface defining the four overridable props for endpoint URLs and URL param key names
- **Filter sync** — URL params are read via `useSearchParams()` and forwarded as query params on outbound API calls, keeping URL state and API state in lockstep
- **`EmptyState`** — Renders contextual empty states: a "Reset Filters" CTA when active filters yield no results, or a generic fallback otherwise
- **`LoadError`** — Canonical error state with retry affordance when either fetch fails

## Usage Example

```typescript
// Zero-config — uses OpenFrame hub defaults
<DeliveryLists />

// Custom endpoints + param keys (must match the consuming chrome's paramKey config)
<DeliveryLists
  completedApiEndpoint="/api/v2/tasks/done"
  inProgressApiEndpoint="/api/v2/tasks/active"
  searchParamKey="q"
  taskTypeParamKey="type"
/>
```

> **Coupling constraint:** `searchParamKey` and `taskTypeParamKey` serve dual roles — they are both the URL param keys this component **reads** and the query param keys it **writes** to outbound API calls. If you override them, ensure the consuming chrome's `section.search.paramKey` / `section.filter.paramKey` and the backend API contract all use the same names.
