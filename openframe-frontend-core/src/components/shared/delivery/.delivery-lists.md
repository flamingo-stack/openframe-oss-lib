Renders the delivery section body with two filterable tables — "Recently Completed" and "Active Tasks" — by fetching from separate API endpoints and reacting to URL search params.

## Key Components

- **`DeliveryLists`** — Main exported component. Reads `search` and `task_type` URL params, fetches both delivery buckets in parallel, and renders two `<DeliveryTable>` instances.
- **`DeliveryListsProps`** — Interface defining the four configurable props (endpoints + param keys).
- **`DEFAULT_COMPLETED_ENDPOINT`** / **`DEFAULT_IN_PROGRESS_ENDPOINT`** — Fallback API routes (`/api/delivery/completed`, `/api/delivery/in-progress`).
- **Coupling constraint** — `searchParamKey` and `taskTypeParamKey` are used for both URL reads (`useSearchParams`) and outbound API query params. Both must stay in sync with the consuming chrome's param keys and the backend API contract.

## Usage Example

```typescript
// Zero-config (OpenFrame defaults)
<DeliveryLists />

// Custom endpoints + param keys (e.g. reverse-proxy setup)
<DeliveryLists
  completedApiEndpoint="/proxy/delivery/done"
  inProgressApiEndpoint="/proxy/delivery/active"
  searchParamKey="q"
  taskTypeParamKey="type"
/>
```

> **Note:** If you override `searchParamKey` or `taskTypeParamKey`, you must also update the consuming chrome's `section.search.paramKey` / `section.filter.paramKey` AND ensure the backend reads the same query string keys. Mismatches between the chrome, this component, and the API will silently break filtering.
