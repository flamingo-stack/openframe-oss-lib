A presentational RAG-search dropdown component that wraps `<SearchInput>` with standardized `<DocSearchResultRow>` rendering, providing a consistent typeahead UI across all doc-search surfaces (data-room sidebar, onboarding guide catalog, etc.).

## Key Components

**`DocSearchBar`** — Main export. A thin presentation wrapper; all data-fetching stays hub-side via `useDocSearch` and is passed in as props.

**`DocSearchBarProps`** — Interface defining the component's contract:
- `query` / `onQueryChange` — controlled input state
- `results` / `isLoading` — hook-fetched search results passed directly
- `onResultSelect` — selection handler preserving modifier keys (`metaKey`, `ctrlKey`, etc.) for new-tab behavior
- `showDropdown` — caller-controlled dropdown visibility override
- `minQueryLength` — typeahead trigger threshold (default: `2`)
- `debounceMs` — input debounce delay (default: `0`; consumers debounce in the hook)
- `renderResult` — optional row renderer override; defaults to `<DocSearchResultRow>`

## Usage Example

```typescript
// Typical consumer — hook lives hub-side, component stays minimal
const { query, setQuery, results, isLoading, handleSelect } = useDocSearch()

return (
  <DocSearchBar
    placeholder="Search docs..."
    query={query}
    onQueryChange={setQuery}
    results={results}
    isLoading={isLoading}
    onResultSelect={handleSelect}
    minQueryLength={2}
  />
)

// Custom row chrome override
<DocSearchBar
  {...baseProps}
  renderResult={(result, isHighlighted) => (
    <MyCustomRow result={result} active={isHighlighted} />
  )}
/>
```

## Design Notes

The hook (`useDocSearch`) intentionally stays hub-side — it depends on hub-only context (`useDocNavigation`, rag-table-config registry, `decideNewTab`). This component accepts hook output as plain props, keeping both consumers at ~5 lines each while the dropdown UI remains identical across surfaces.
