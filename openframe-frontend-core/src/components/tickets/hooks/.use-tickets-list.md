Fetches and paginates a customer's own ticket list by wrapping `POST /api/chat/agent/find-ticket`, returning tickets, pagination metadata, and loading state via TanStack Query.

## Key Components

**`useTicketsList(filters)`** — Primary export. Builds a cache-keyed TanStack Query that posts to the find-ticket endpoint with search, status, and pagination params. Intentionally sets `staleTime: 0` / `gcTime: 0` / `refetchOnMount: 'always'` to ensure the customer always sees live ticket data.

**`UseTicketsListFilters`** — Input interface:
- `customerEmail` — required; disables the query if empty (prevents anon 401 round-trips)
- `search` — free-text FTS query; empty string means "list all my tickets"
- `status` — `'open'` | `'closed'` | `'all'` / empty (no filter)
- `page` / `pageSize` — 1-based pagination; server caps `pageSize` at 100
- `refetchInterval` — optional polling interval (ms) for live status updates; pauses on hidden tabs

**`UseTicketsListReturn`** — Output interface including `tickets`, `isLoading`, `isFetching`, `error`, `refetch`, `lastUpdatedAt`, `totalCount`, `page`, `pageSize`, and pre-computed `totalPages`.

**Cache key** — `['tickets', 'self', identityKey, search, statusFilter, page, pageSize]`. Each filter+page combo occupies its own slot; mutations via `useTicketActions` invalidate all `['tickets']` slots together.

## Usage Example

```typescript
const {
  tickets,
  isLoading,
  totalCount,
  totalPages,
  page,
  refetch,
} = useTicketsList({
  customerEmail: identity.user.email,
  search: searchQuery,
  status: 'open',
  page: currentPage,
  pageSize: 20,
  refetchInterval: isDrawerOpen ? 5000 : false,
})
```

> **Note:** `customerEmail` must be resolved by the parent (e.g. via `useChatIdentity`) before mounting — drilling it in prevents a race condition where calling `useChatIdentity` inside this hook would produce a `null` identity on the first render, causing an empty-state flash.
