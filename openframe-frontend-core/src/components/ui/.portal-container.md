Provides a React context for overriding the default Radix UI portal target, allowing overlay components (dropdowns, tooltips) to portal into a specific DOM node instead of `document.body`.

## Key Components

- **`PortalContainerContext`** — React context holding the target `HTMLElement` (or `null` to fall back to `document.body`)
- **`usePortalContainer()`** — Hook that returns the active portal container from context

## Usage Example

```typescript
// Wrap a high-z surface (e.g. chat drawer) with a ref-bound provider
// so all Radix overlays inside it inherit the local stacking context.

function ChatDrawer() {
  const [container, setContainer] = React.useState<HTMLElement | null>(null)

  return (
    <PortalContainerContext.Provider value={container}>
      <div ref={setContainer} className="chat-drawer">
        <DropdownMenu>
          {/* Portal targets the drawer div, not document.body */}
          <DropdownMenuContent />
        </DropdownMenu>
      </div>
    </PortalContainerContext.Provider>
  )
}

// Inside any Radix component, pass the container to the Portal prop
function MyDropdown() {
  const portalContainer = usePortalContainer()

  return (
    <DropdownMenu.Portal container={portalContainer}>
      <DropdownMenu.Content>...</DropdownMenu.Content>
    </DropdownMenu.Portal>
  )
}
```

> **Why not `document.body`?** Portaling to `document.body` breaks z-index isolation — overlays inside a high-z surface must compete at the document root. A scoped portal node keeps z-indices small and local. Radix uses `position: fixed` internally, so `overflow: hidden` clipping is still escaped regardless of the portal target.
