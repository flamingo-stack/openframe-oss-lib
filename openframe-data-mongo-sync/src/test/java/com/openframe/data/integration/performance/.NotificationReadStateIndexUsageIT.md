Integration test that validates MongoDB index usage and query performance for the `NotificationReadState` collection, ensuring that common notification read-state operations (list, mark read, unread counts, etc.) use covered index plans and stay within defined latency budgets.

## Key Components

| Component | Description |
|-----------|-------------|
| `NotificationReadStateIndexUsageIT` | Main test class extending `BaseMongoIntegrationTest`; tagged `performance`, enabled only when `-Dperformance.tests=true` |
| `seed()` / `@BeforeAll` | Drops and re-seeds `notifications` + `notification_read_states` collections with configurable volume (default 30k notifications, 3k hot-admin read states) |
| `MongoExplain` | Helper that runs `explain()` on find, upsert, and aggregation queries and returns a `Stats` object for assertions |
| `PerfResultRecorder` | Writes per-test timing/key-examined metrics to a Markdown report under `target/perf-results/` |

### Indexes Under Test

| Index Constant | Purpose |
|----------------|---------|
| `recipient_notification_unique` | Unique compound — recipient + notificationId; covers `markRead` single-key lookups |
| `recipient_status` | Covers `hasUnread`, `markAllAsRead`, pagination step 1 |
| `recipient_category_status` | Covers `unreadCountsByCategory` aggregation `$match` stage |

## Usage Example

```bash
# Run the performance suite (requires a live MongoDB instance wired via BaseMongoIntegrationTest)
mvn verify -Dperformance.tests=true \
           -Dperf.notifications.count=30000 \
           -Dperf.readStates.hotAdmin=3000 \
           -Dperf.readStates.hotMachine=1000 \
           -Dperf.readStates.noise=20000
```

```java
// Inline assertion pattern used by each test
Stats stats = MongoExplain.explainFind(mongoTemplate, READS_COLL, query);

stats.assertNoCollectionScan()
     .assertUsesIndex(IDX_RECIPIENT_STATUS)
     .assertExecutionTimeBelow(FIND_BUDGET_MS)  // 200 ms
     .assertExaminedAtMost(2);
```

> Performance budgets: **200 ms** for reads, **100 ms** for writes. Results are emitted to `target/perf-results/<TestClass>.md` after each run.
