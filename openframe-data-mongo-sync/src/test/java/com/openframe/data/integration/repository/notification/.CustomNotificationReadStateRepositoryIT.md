Integration test suite for `CustomNotificationReadStateRepository`, verifying the `bulkInsertUnordered` operation against a live MongoDB instance.

## Key Components

- **`CustomNotificationReadStateRepositoryIT`** — Spring Boot integration test class extending `BaseMongoIntegrationTest`, gated by the `integration.tests` system property
- **`resetCollections()`** — Drops and re-indexes `Notification` and `NotificationReadState` collections before each test for isolation
- **`row()`** — Factory helper that builds a `NotificationReadState` with `UNREAD` status and `TICKETS` category for a given recipient/notification pair

## Test Cases

| Test | Scenario |
|------|----------|
| `bulk_insert_persists_all_rows` | Two rows for different recipients on one notification are both persisted |
| `empty_input_noop` | Empty list input is a silent no-op — no exception, no rows written |
| `duplicate_keys_swallowed_only_new_rows_added` | Re-inserting existing keys in `UNORDERED` mode silently skips duplicates; only new rows land |
| `full_overlap_idempotent` | Full duplicate batch leaves row count unchanged — broadcast retries are safe |

## Usage Example

```java
// Run integration tests with the required system property
./mvnw verify -Dintegration.tests=true

// Or target just this class
./mvnw test -Dtest=CustomNotificationReadStateRepositoryIT -Dintegration.tests=true
```

> **Key contract verified:** `bulkInsertUnordered` uses MongoDB's unordered bulk write mode, meaning duplicate-key errors on individual documents are swallowed without aborting the batch. This makes broadcast notification fan-out retries fully idempotent.
