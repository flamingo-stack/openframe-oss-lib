Manages the read/unread state lifecycle for notifications, handling creation, status updates, deletions, and unread counts per recipient.

## Key Components

| Method | Description |
|--------|-------------|
| `createForAudience` | Bulk-inserts `NotificationReadState` records (status `UNREAD`) for a list of recipients |
| `hasUnread` | Returns `true` if a recipient has any unread notifications |
| `markRead` | Marks a single notification as read for a specific recipient |
| `markAllAsRead` | Marks all notifications as read for a recipient, returns affected count |
| `deleteNotification` | Soft-deletes a single notification for a recipient |
| `deleteAllRead` | Soft-deletes all read notifications for a recipient, returns affected count |
| `unreadCountsByCategory` | Returns a `Map<NotificationCategory, Long>` of unread counts grouped by category for the current tenant |

## Usage Example

```java
// Fan out a new notification to multiple technicians
notificationReadStateService.createForAudience(
    "notif-001",
    NotificationCategory.ALERT,
    "Disk usage critical",
    RecipientType.TECHNICIAN,
    List.of("tech-1", "tech-2", "tech-3")
);

// Check if a user has unread items before rendering the badge
boolean showBadge = notificationReadStateService.hasUnread("tech-1", RecipientType.TECHNICIAN);

// Mark a single notification read on click
notificationReadStateService.markRead("tech-1", RecipientType.TECHNICIAN, "notif-001");

// Get per-category unread counts for the notification panel
Map<NotificationCategory, Long> counts =
    notificationReadStateService.unreadCountsByCategory("tech-1", RecipientType.TECHNICIAN);
// e.g. { ALERT: 3, TICKET: 7 }
```

> **Note:** Deletions are soft-deletes — records are flagged rather than removed, preserving audit history. All methods are Bean Validation–enforced via `@Validated`.
