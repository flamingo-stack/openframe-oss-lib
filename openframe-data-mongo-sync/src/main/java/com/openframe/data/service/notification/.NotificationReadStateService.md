Manages the read/unread state lifecycle for notifications, handling creation, status updates, deletions, and unread counts per recipient.

## Key Components

| Method | Description |
|--------|-------------|
| `createForAudience` | Bulk-inserts `NotificationReadState` records (status `UNREAD`) for a collection of recipients |
| `hasUnread` | Returns `true` if a recipient has any unread notifications |
| `markRead` | Marks a single notification as read for a specific recipient |
| `markAllAsRead` | Marks all notifications as read for a recipient, returns affected count |
| `deleteNotification` | Soft-deletes a single notification entry for a recipient |
| `deleteAllRead` | Soft-deletes all read notifications for a recipient, returns affected count |
| `unreadCountsByCategory` | Returns a `Map<NotificationCategory, Long>` of unread counts grouped by category for the current tenant |

## Usage Example

```java
// Fan out a new notification to a set of users
notificationReadStateService.createForAudience(
    "notif-001",
    NotificationCategory.ALERT,
    "Disk usage critical",
    RecipientType.TECHNICIAN,
    List.of("user-1", "user-2", "user-3")
);

// Check for unread badge indicator
boolean hasBadge = notificationReadStateService.hasUnread("user-1", RecipientType.TECHNICIAN);

// Mark one notification read
notificationReadStateService.markRead("user-1", RecipientType.TECHNICIAN, "notif-001");

// Get per-category unread counts (e.g. for sidebar counters)
Map<NotificationCategory, Long> counts =
    notificationReadStateService.unreadCountsByCategory("user-1", RecipientType.TECHNICIAN);
// { ALERT -> 3, TICKET -> 7 }

// Bulk-clear all read entries
notificationReadStateService.deleteAllRead("user-1", RecipientType.TECHNICIAN);
```

> Soft-delete methods preserve audit history while removing entries from active views. All methods are validated via Jakarta Bean Validation — `null` or blank arguments will throw a constraint violation before reaching the repository layer.
