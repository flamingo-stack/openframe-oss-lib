Manages persistence and lifecycle of `OpenFrameClientConfiguration` documents, including version tracking, publish state transitions, and download configuration merging.

## Key Components

| Method | Description |
|---|---|
| `get()` | Retrieves the singleton default configuration (ID: `"default"`), throwing if absent |
| `findById(String id)` | Optional lookup by arbitrary configuration ID |
| `save(OpenFrameClientConfiguration)` | Persists a configuration document via the repository |
| `markAsNonPublished(config)` | Transitions publish state to non-published using current state |
| `markAsPublished(config)` | Transitions publish state to published |
| `updateVersion(String newVersion)` | Updates the client version and marks config as `pending` publish; no-ops if version unchanged. Retries on optimistic locking failure |
| `updateConfigurationFields(config)` | Upserts a configuration by ID; merges download config and flags `pending` if changed. Retries on optimistic locking failure |
| `mergeAndSave(existing, fromConfig)` | Internal merge: updates `downloadConfiguration` on the existing document and triggers pending state if a diff is detected |

## Usage Example

```java
// Retrieve and update the default client configuration version
openFrameClientConfigurationService.updateVersion("1.4.2");

// Merge incoming configuration (e.g. from a remote sync)
OpenFrameClientConfiguration incoming = new OpenFrameClientConfiguration();
incoming.setId("default");
incoming.setDownloadConfiguration(newDownloadConfigs);
openFrameClientConfigurationService.updateConfigurationFields(incoming);

// Manually transition publish state
OpenFrameClientConfiguration config = openFrameClientConfigurationService.get();
openFrameClientConfigurationService.markAsPublished(config);
```

## Notes

- The constant `DEFAULT_ID = "default"` is the expected singleton document ID in the repository.
- `@RetryOnOptimisticLockingFailure` on `updateVersion` and `updateConfigurationFields` ensures safe concurrent writes without manual locking.
- Publish state changes are only triggered when actual data differences are detected, avoiding unnecessary re-publishes.
