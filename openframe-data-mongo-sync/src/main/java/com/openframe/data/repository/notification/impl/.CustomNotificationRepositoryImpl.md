Custom MongoDB repository implementation for paginated notification queries, supporting cursor-based pagination, read-status filtering, and title search for a given recipient.

## Key Components

| Component | Description |
|-----------|-------------|
| `findPageForRecipient` | Public entry point; orchestrates read-state fetch and notification hydration |
| `fetchReadStates` | Queries `NotificationReadState` collection with filtering, cursor, and sort; returns an ordered `notificationId → ReadStatus` map |
| `fetchNotificationsPreservingOrder` | Bulk-fetches `Notification` documents by ID, then re-orders results to match the cursor-page order |
| `applyStatusFilter` | Excludes `DELETED` records by default; filters to `READ` or `UNREAD` when a boolean filter is supplied |
| `applyCursor` | Appends `lt`/`gt` predicate on `notificationId` to implement keyset (cursor) pagination in both forward and backward directions |
| `applySearch` | Applies a case-insensitive regex on the `title` field using `Pattern.quote` to prevent injection |

## Usage Example

```java
// Injected via Spring; called from a service layer
NotificationPage page = customNotificationRepository.findPageForRecipient(
    "user-123",          // recipientId
    RecipientType.USER,  // recipientType
    false,               // readFilter: null=all, true=read, false=unread
    "server alert",      // search term (title contains)
    lastSeenId,          // cursor from previous page
    false,               // backward=false → newest-first
    20                   // page size
);

List<NotificationWithStatus> notifications = page.getItems();
```

> **Design note:** The two-phase fetch (read-state first, then notifications) keeps the pagination index lean and avoids a join, while `LinkedHashMap` preserves the cursor-sorted order across the document lookup.
