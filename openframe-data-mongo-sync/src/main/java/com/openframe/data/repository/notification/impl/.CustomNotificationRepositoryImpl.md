Custom MongoDB repository implementation for paginated, cursor-based notification queries. Handles filtering by recipient, read status, search term, and sort direction using a two-phase fetch strategy (read states first, then full notification documents).

## Key Components

| Method | Description |
|---|---|
| `findPageForRecipient(...)` | Public entry point; returns a `NotificationPage` for a given recipient with optional filters and cursor pagination |
| `fetchReadStates(...)` | Phase 1 — queries `NotificationReadState` collection with all filters applied; returns an ordered map of `notificationId → ReadStatus` |
| `fetchNotificationsPreservingOrder(...)` | Phase 2 — bulk-fetches `Notification` documents by ID, then re-orders them to match the read-state result order |
| `effectiveDirection(...)` | Reverses sort direction when paginating backward |
| `applyStatusFilter(...)` | Adds a `status` criteria: excludes `DELETED` when filter is `null`, or matches `READ`/`UNREAD` exactly |
| `applyCursor(...)` | Adds a `lt`/`gt` bound on `notificationId` for keyset pagination |
| `applySearch(...)` | Applies a case-insensitive regex filter on the `title` field |

## Usage Example

```java
// Fetch the first page of unread notifications, newest first
NotificationPage page = repository.findPageForRecipient(
    "user-123",
    RecipientType.USER,
    false,          // unread only
    "alert",        // search term
    null,           // no cursor (first page)
    false,          // forward pagination
    Sort.Direction.DESC,
    20              // page size
);

// Fetch the next page using the last seen notification ID as cursor
NotificationPage nextPage = repository.findPageForRecipient(
    "user-123",
    RecipientType.USER,
    false,
    "alert",
    "64f1a2b3c4d5e6f7a8b9c0d1", // cursor from previous page
    false,
    Sort.Direction.DESC,
    20
);
```

> **Note:** The two-phase fetch (read states → notifications) keeps the sort and filter logic on the lightweight `NotificationReadState` collection, then resolves full document content only for the final result set, avoiding expensive joins at the MongoDB level.
