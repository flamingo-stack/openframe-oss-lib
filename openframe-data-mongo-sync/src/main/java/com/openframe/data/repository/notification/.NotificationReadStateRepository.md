MongoDB repository interface for managing notification read state persistence, providing CRUD operations and bulk status update capabilities for tracking per-recipient notification read/unread/deleted states.

## Key Components

| Method | Description |
|--------|-------------|
| `existsByRecipientIdAndRecipientTypeAndStatus` | Checks if a read state record exists for a given recipient and status |
| `markAsRead` | Atomically marks a single notification as `READ` with a timestamp for a specific recipient |
| `markAllAsRead` | Bulk-marks all `UNREAD` notifications as `READ` for a recipient |
| `softDelete` | Soft-deletes a single notification by setting status to `DELETED` (skips already-deleted records) |
| `softDeleteAllRead` | Bulk soft-deletes all `READ` notifications for a recipient |
| `unreadCountsByCategory` | Aggregation query returning unread notification counts grouped by category for a tenant-scoped recipient |

## Usage Example

```java
// Mark a single notification as read
long updated = repository.markAsRead("user-123", RecipientType.USER, "notif-456");

// Mark all unread notifications as read for a recipient
long count = repository.markAllAsRead("user-123", RecipientType.USER);

// Soft-delete a specific notification
long deleted = repository.softDelete("user-123", RecipientType.USER, "notif-456");

// Bulk soft-delete all read notifications
long cleaned = repository.softDeleteAllRead("user-123", RecipientType.USER);

// Get unread counts grouped by category
List<CategoryCount> counts = repository.unreadCountsByCategory(
    "user-123",
    RecipientType.USER,
    "tenant-001"
);

// Check for any unread notifications
boolean hasUnread = repository.existsByRecipientIdAndRecipientTypeAndStatus(
    "user-123",
    RecipientType.USER,
    ReadStatus.UNREAD
);
```

> All status mutations use MongoDB `$set` with `$$NOW` for server-side timestamping, ensuring atomic updates without read-modify-write cycles. Also extends `CustomNotificationReadStateRepository` for additional query flexibility.
