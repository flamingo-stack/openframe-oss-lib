MongoDB repository interface for managing notification read state persistence, providing CRUD operations and bulk update capabilities for tracking per-recipient notification status.

## Key Components

| Method | Description |
|--------|-------------|
| `existsByRecipientIdAndRecipientTypeAndStatus` | Checks if any notification matching the given recipient and status exists |
| `markAsRead` | Marks a single notification as `READ` for a specific recipient, setting `readAt` to current timestamp |
| `markAllAsRead` | Bulk-marks all `UNREAD` notifications as `READ` for a recipient |
| `softDelete` | Soft-deletes a single notification by setting its status to `DELETED` (excludes already-deleted records) |
| `softDeleteAllRead` | Soft-deletes all `READ` notifications for a recipient in bulk |
| `unreadCountsByCategory` | Aggregates unread notification counts grouped by `category` for a given recipient and tenant |

## Usage Example

```java
// Check for any unread notifications
boolean hasUnread = repository.existsByRecipientIdAndRecipientTypeAndStatus(
    "user-123", RecipientType.USER, ReadStatus.UNREAD
);

// Mark a single notification as read
long updated = repository.markAsRead("user-123", RecipientType.USER, "notif-456");

// Mark all notifications as read
long totalMarked = repository.markAllAsRead("user-123", RecipientType.USER);

// Soft delete a specific notification
long deleted = repository.softDelete("user-123", RecipientType.USER, "notif-456");

// Get unread counts per category
List<CategoryCount> counts = repository.unreadCountsByCategory(
    "user-123", RecipientType.USER, "tenant-789"
);
```

## Notes

- All write operations use MongoDB `@Update` to perform atomic in-place updates without full document replacement
- `softDelete` guards against double-deletion via `$ne: DELETED` filter
- `unreadCountsByCategory` uses a two-stage aggregation pipeline (`$match` → `$group`) scoped to a specific tenant
