Custom repository interface for cursor-based paginated notification queries, supporting filtering by recipient, read status, and search terms.

## Key Components

- **`findPageForRecipient` (abstract)** — Core paginated query method accepting full parameter control including sort direction, cursor position, and traversal direction for bi-directional pagination.
- **`findPageForRecipient` (default)** — Convenience overload that defaults sort direction to `DESC`, simplifying the most common use case.

## Parameters

| Parameter | Type | Description |
|---|---|---|
| `recipientId` | `String` | Unique identifier of the notification recipient |
| `recipientType` | `RecipientType` | Enum distinguishing recipient categories (e.g. user, tenant) |
| `readFilter` | `Boolean` | Nullable filter for read/unread status; `null` returns all |
| `search` | `String` | Optional text search against notification content |
| `cursor` | `String` | Opaque pagination cursor from a previous page result |
| `backward` | `boolean` | `true` to paginate backwards from the cursor |
| `direction` | `Sort.Direction` | Sort order; defaults to `DESC` in the convenience overload |
| `limit` | `int` | Maximum number of notifications per page |

## Usage Example

```java
// Using the default overload (DESC order)
NotificationPage page = notificationRepository.findPageForRecipient(
    "user-123",
    RecipientType.USER,
    false,        // unread only
    "alert",      // search term
    null,         // no cursor = first page
    false,        // forward pagination
    20            // page size
);

// Using the full overload with explicit sort direction
NotificationPage page = notificationRepository.findPageForRecipient(
    "tenant-456",
    RecipientType.TENANT,
    null,
    null,
    "cursor-opaque-token",
    false,
    Sort.Direction.ASC,
    10
);
```
