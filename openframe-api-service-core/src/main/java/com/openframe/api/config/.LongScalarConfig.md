Custom GraphQL scalar configuration that maps Java `Long` (64-bit) values to a GraphQL `Long` scalar type, enabling support for large numbers that exceed the standard GraphQL `Int` limit of ~2.1 billion (e.g., file sizes in bytes).

## Key Components

| Method | Description |
|---|---|
| `serialize` | Converts Java `Long` or `Number` instances to `Long` for GraphQL responses |
| `parseValue` | Parses variables from `Long`, `Number`, or `String` inputs into `Long` |
| `parseLiteral` | Converts GraphQL AST literals (`IntValue`, `StringValue`) into `Long` |

## Usage Example

```java
// GraphQL Schema Definition
scalar Long

type File {
  id: ID!
  name: String!
  sizeInBytes: Long   # Supports values > 2,147,483,647
}

// Java Entity
public class FileEntity {
    private Long sizeInBytes; // e.g., 10_000_000_000L (10 GB)
}
```

```java
// Bean is auto-registered via @DgsScalar + @Component — no manual wiring needed
// Accepted input formats in GraphQL queries:

// As integer literal:
{ file(id: "1") { sizeInBytes } }  // returns Long

// As variable:
query($size: Long!) { ... }        // accepts Number or "12345678901" string
```

> **Why `Long` instead of `Int`?** GraphQL's built-in `Int` is a signed 32-bit integer (max ~2.1 GB). File sizes, timestamps in milliseconds, and other large numeric values require 64-bit precision, handled by this scalar per the [IBM Long Scalar Spec](https://ibm.github.io/graphql-specs/custom-scalars/long.html).
