Centralized GraphQL exception handler that intercepts data fetcher errors and maps them to structured `GraphQLError` responses with standardized error codes and metadata.

## Key Components

### `GraphQLExceptionHandler`
Extends `SimpleDataFetcherExceptionHandler` and overrides `handleException` to provide typed error mapping:

| Exception Type | Error Code | Behavior |
|---|---|---|
| `PinotQueryException` | `PINOT_QUERY_ERROR` | Generic query failure message |
| `DataAccessException` | `DATABASE_ERROR` | Generic DB failure message |
| `NotFoundException` | From exception | Passes through original message |
| `ConflictException` | From exception | Passes through original message |
| `BaseException` | From exception | Passes through original message |
| `IllegalArgumentException` / `IllegalStateException` | `VALIDATION_ERROR` | Passes through original message |
| `RuntimeException` / fallback | `INTERNAL_ERROR` | Generic unexpected error message |

### `buildError(String message, ErrorCode errorCode)`
Private helper that constructs a `GraphQLError` with a consistent extension payload:
- `code` — machine-readable error code
- `httpStatus` — corresponding HTTP status
- `timestamp` — epoch milliseconds of the error

## Usage Example

```java
// Registered automatically as a Spring @Component.
// GraphQL Java picks it up via DataFetcherExceptionHandler SPI.

// Example GraphQL error response extensions:
// {
//   "errors": [{
//     "message": "Resource not found",
//     "extensions": {
//       "code": "NOT_FOUND",
//       "httpStatus": 404,
//       "timestamp": 1718000000000
//     }
//   }]
// }
```

> All exceptions are logged at `ERROR` level before mapping, ensuring full stack traces are captured server-side while clients receive safe, sanitized messages.
