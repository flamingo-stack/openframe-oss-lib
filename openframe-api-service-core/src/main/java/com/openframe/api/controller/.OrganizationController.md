REST API controller handling organization mutation operations (Create, Update, Delete, Status changes) mapped to `/organizations`. Read operations are handled separately in the external API layer.

## Key Components

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/organizations` | Create a new organization |
| `PUT` | `/organizations/{id}` | Update an existing organization |
| `GET` | `/organizations/{id}/can-archive` | Check if organization is eligible for archiving |
| `PATCH` | `/organizations/{id}/status` | Update organization status (`ACTIVE` / `ARCHIVED`) |

**Dependencies:**
- `OrganizationService` — read/validation operations (e.g., archive eligibility check)
- `OrganizationCommandService` — write operations (create, update, status change)
- `OrganizationMapper` — maps domain entities to `OrganizationResponse` DTOs

## Usage Example

```java
// Create a new organization
POST /organizations
Content-Type: application/json

{
  "name": "Acme Corp"
}
// → 201 Created: OrganizationResponse

// Update an existing organization
PUT /organizations/org-123
Content-Type: application/json

{
  "name": "Acme Corporation"
}
// → 200 OK: OrganizationResponse

// Check archive eligibility
GET /organizations/org-123/can-archive
// → 200 OK: true | false

// Archive an organization (409 Conflict if active devices exist)
PATCH /organizations/org-123/status
Content-Type: application/json

{
  "status": "ARCHIVED"
}
// → 204 No Content
```

> **Note:** Archiving an organization with active linked devices returns `409 Conflict`. Use `GET /organizations/{id}/can-archive` to validate eligibility before patching status. `PUT /{id}` returns `404 Not Found` if the organization does not exist.
