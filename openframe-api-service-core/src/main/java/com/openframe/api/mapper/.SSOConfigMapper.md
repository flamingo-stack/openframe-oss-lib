Maps between SSO configuration DTOs and the `SSOConfig` document entity, handling client secret encryption during entity creation and updates.

## Key Components

| Method | Description |
|--------|-------------|
| `toEntity()` | Creates a new `SSOConfig` document from a provider identifier and request DTO, encrypting the client secret via `EncryptionService` |
| `updateEntity()` | Applies changes from a request DTO to an existing `SSOConfig`, re-encrypting the client secret |
| `toResponse()` | Builds a `SSOConfigResponse` DTO from a `SSOConfig` entity, accepting a pre-decrypted secret string |

## Usage Example

```java
@Autowired
private SSOConfigMapper ssoConfigMapper;

@Autowired
private EncryptionService encryptionService;

// Create a new SSO config entity
SSOConfig newConfig = ssoConfigMapper.toEntity("microsoft", request, encryptionService);

// Update an existing SSO config entity
ssoConfigMapper.updateEntity(existingConfig, request, encryptionService);

// Map entity to response DTO with decrypted secret
String decryptedSecret = encryptionService.decryptClientSecret(config.getClientSecret());
SSOConfigResponse response = ssoConfigMapper.toResponse(config, decryptedSecret);
```

## Notes

- Client secrets are **always encrypted** on write (`toEntity`, `updateEntity`) and must be decrypted externally before passing to `toResponse`
- `autoProvisionUsers` defaults to `false` when the request value is `null` (safe `TRUE.equals()` null check)
- New entities are created with `enabled = true` by default; the update path does not mutate the `enabled` flag
