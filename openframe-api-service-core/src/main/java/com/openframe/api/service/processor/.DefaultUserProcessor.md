Default fallback implementation of the `UserProcessor` interface, activated automatically via `@ConditionalOnMissingBean` when no custom `UserProcessor` bean is present in the application context.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `postProcessUserDeleted(User)` | Method | Logs debug message with the deleted user's ID |
| `postProcessUserGet(UserPageResponse)` | Method | Logs debug message with the count of fetched users |
| `postProcessUserGet(UserResponse)` | Method | Logs debug message with the fetched user's ID |
| `postProcessUserUpdated(User)` | Method | Logs debug message with the updated user's ID |

## Usage Example

This bean is auto-registered unless overridden. To provide custom post-processing logic, define your own `UserProcessor` implementation:

```java
@Component
public class CustomUserProcessor implements UserProcessor {

    @Override
    public void postProcessUserDeleted(User user) {
        // Custom logic — e.g., audit trail, cleanup, notifications
        auditService.recordDeletion(user.getId());
    }

    @Override
    public void postProcessUserGet(UserPageResponse response) {
        // Custom logic — e.g., enrich or filter page results
    }

    @Override
    public void postProcessUserGet(UserResponse response) {
        // Custom logic — e.g., apply role-based field masking
    }

    @Override
    public void postProcessUserUpdated(User user) {
        // Custom logic — e.g., sync to external directory
    }
}
```

> When a custom `UserProcessor` bean is detected, Spring skips `DefaultUserProcessor` entirely due to the `@ConditionalOnMissingBean` annotation — no additional configuration required.
