Manages SSO (Single Sign-On) provider configurations, handling CRUD operations, enable/disable toggling, and validation logic for auto-provisioning rules across identity providers.

## Key Components

| Method | Description |
|--------|-------------|
| `getEnabledProviders()` | Returns a list of currently enabled SSO providers for use by login UI components |
| `getAvailableProviders()` | Returns all configured SSO providers from application properties |
| `getConfig(provider)` | Retrieves full SSO config for a provider, with decrypted client secret; returns a safe empty response if not found |
| `upsertConfig(provider, request)` | Creates or updates an SSO configuration, triggering post-processing hooks |
| `deleteConfig(provider)` | Removes an SSO configuration and fires post-delete processor hooks |
| `toggleEnabled(provider, enabled)` | Enables or disables a provider without modifying other settings |
| `validateAutoProvision(provider, request)` | Enforces domain and tenant rules when auto-provisioning is requested |

## Usage Example

```java
// Fetch enabled providers for login page rendering
List<SSOConfigStatusResponse> providers = ssoConfigService.getEnabledProviders();

// Upsert a Microsoft SSO configuration
SSOConfigRequest request = SSOConfigRequest.builder()
    .clientId("my-client-id")
    .clientSecret("my-secret")
    .msTenantId("tenant-123")
    .autoProvisionUsers(true)
    .allowedDomains(List.of("example.com"))
    .build();

SSOConfigResponse response = ssoConfigService.upsertConfig("microsoft", request);

// Toggle provider off
ssoConfigService.toggleEnabled("microsoft", false);
```

## Validation Notes

- Auto-provisioning requires at least one `allowedDomains` entry
- Microsoft provider with auto-provisioning also requires a non-blank `msTenantId`
- Domains are normalized (trimmed, lowercased, deduplicated) before persistence and validated against public/generic domain rules via `DomainValidationService`
