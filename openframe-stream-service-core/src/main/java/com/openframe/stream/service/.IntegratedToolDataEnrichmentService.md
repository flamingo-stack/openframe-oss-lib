Enriches deserialized Debezium messages from integrated tools with machine and tenant metadata by resolving agent IDs and cluster identifiers against cached data sources.

## Key Components

| Component | Description |
|-----------|-------------|
| `getExtraParams(DeserializedDebeziumMessage)` | Main enrichment entry point; populates an `IntegratedToolEnrichedData` object with machine and tenant context |
| `enrichFromMachine()` | Resolves `agentId` → `CachedMachineInfo` (machineId, hostname) and `CachedOrganizationInfo` (organizationId, name) via Redis cache |
| `enrichFromTenant()` | Resolves `tenantId` using either `TenantIdProvider` (single-tenant clusters) or `ClusterTenantIdResolver` (shared/multi-tenant clusters) |
| `getType()` | Returns `DataEnrichmentServiceType.INTEGRATED_TOOLS_EVENTS` to identify this enricher in the service registry |
| `ClusterTenantIdResolver` | Optional bean (`required = false`); when absent, falls back to `TenantIdProvider` for single-tenant deployments |

## Usage Example

```java
// Injected automatically by Spring; consumed by a Kafka/Debezium stream processor
@Autowired
private IntegratedToolDataEnrichmentService enrichmentService;

DeserializedDebeziumMessage message = kafkaConsumer.poll();

// Returns enriched metadata without mutating the original message fields
// (except tenantId, which is set back on the message)
IntegratedToolEnrichedData enriched = enrichmentService.getExtraParams(message);

log.info("Machine: {}, Org: {}, Tenant: {}",
    enriched.getMachineId(),
    enriched.getOrganizationName(),
    enriched.getTenantId());
```

## Notes

- `MachineIdCacheService` lookups are Redis-backed; a cache miss logs a `WARN` and skips machine/org enrichment gracefully.
- `ClusterTenantIdResolver` is optional — omit the bean for single-tenant cluster deployments where every event belongs to the same tenant.
- Both `enriched.getTenantId()` and `message.getTenantId()` are updated in `enrichFromTenant()` to keep downstream consumers consistent.
