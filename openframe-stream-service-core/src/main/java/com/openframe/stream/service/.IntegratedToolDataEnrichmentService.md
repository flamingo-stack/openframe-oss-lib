Enriches deserialized Debezium CDC messages from integrated tools with machine and tenant metadata by resolving agent IDs and cluster identifiers against cached data sources.

## Key Components

| Component | Description |
|-----------|-------------|
| `getExtraParams(DeserializedDebeziumMessage)` | Entry point that returns an `IntegratedToolEnrichedData` object populated with machine and tenant context |
| `enrichFromMachine(...)` | Resolves `agentId` → `CachedMachineInfo` (machineId, hostname) and `organizationId`/`organizationName` via Redis cache |
| `enrichFromTenant(...)` | Resolves tenantId using either `TenantIdProvider` (dedicated tenant clusters) or `ClusterTenantIdResolver` (shared clusters mapping domain → canonical tenantId) |
| `getType()` | Returns `DataEnrichmentServiceType.INTEGRATED_TOOLS_EVENTS`, identifying this service in the enrichment pipeline |
| `ClusterTenantIdResolver` | Optional bean (`required = false`); when absent, the cluster is treated as single-tenant |

## Usage Example

```java
// Injected automatically by Spring in the enrichment pipeline
@Autowired
IntegratedToolDataEnrichmentService enrichmentService;

DeserializedDebeziumMessage message = kafkaConsumer.poll();

// Returns enriched metadata; also mutates message.tenantId in-place
IntegratedToolEnrichedData enriched = enrichmentService.getExtraParams(message);

log.info("machineId={}, hostname={}, org={}, tenant={}",
    enriched.getMachineId(),
    enriched.getHostname(),
    enriched.getOrganizationName(),
    enriched.getTenantId()
);
```

## Notes

- `MachineIdCacheService` is backed by Redis; a cache miss logs a `WARN` and skips machine-level fields.
- `ClusterTenantIdResolver` is **optional** — omit the bean for dedicated single-tenant cluster deployments (e.g., MeshCentral `domain`-based multi-tenant setups require it).
- Both `enriched.tenantId` and `message.tenantId` are updated in `enrichFromTenant`, keeping the source message consistent for downstream processors.
