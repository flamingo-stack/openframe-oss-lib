A technology-agnostic base interface defining common user repository operations, supporting both blocking (`Optional`/`boolean`) and reactive (`Mono<T>`/`Mono<Boolean>`) implementations through generic type parameters.

## Key Components

| Generic | Blocking Type | Reactive Type | Description |
|---------|--------------|---------------|-------------|
| `T` | `Optional<User>` | `Mono<User>` | Return type wrapper |
| `B` | `boolean` | `Mono<Boolean>` | Boolean return type |
| `ID` | `String` | `String` | Entity identifier type |

### Methods

| Method | Description |
|--------|-------------|
| `findByEmail(String email)` | Looks up a user by their email address |
| `existsByEmail(String email)` | Checks whether a user with the given email exists |
| `existsByEmailAndStatus(String email, UserStatus status)` | Checks whether a user exists with a specific email and `UserStatus` |

## Usage Example

```java
// Blocking implementation
public interface UserRepository
    extends BaseUserRepository<Optional<User>, Boolean, String> {
    // inherits findByEmail, existsByEmail, existsByEmailAndStatus
}

// Reactive implementation
public interface ReactiveUserRepository
    extends BaseUserRepository<Mono<User>, Mono<Boolean>, String> {
    // inherits findByEmail, existsByEmail, existsByEmailAndStatus
}

// Blocking usage
Optional<User> user = userRepository.findByEmail("admin@example.com");
boolean exists = userRepository.existsByEmailAndStatus("admin@example.com", UserStatus.ACTIVE);

// Reactive usage
Mono<User> user = reactiveUserRepository.findByEmail("admin@example.com");
Mono<Boolean> exists = reactiveUserRepository.existsByEmailAndStatus("admin@example.com", UserStatus.ACTIVE);
```

> Blocking repositories should override `existsByEmail` since the default implementation targets reactive consumers.
