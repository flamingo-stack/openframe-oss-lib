Generic repository interface for querying integrated tools across both blocking and reactive implementations, with built-in support for tenant-aware queries.

## Key Components

| Element | Description |
|---------|-------------|
| `BaseIntegratedToolRepository<T, B, ID>` | Generic base interface parameterized for blocking vs. reactive use |
| `T` | Return type wrapper — `Optional<IntegratedTool>` (blocking) or `Mono<IntegratedTool>` (reactive) |
| `B` | Boolean return type — `boolean` (blocking) or `Mono<Boolean>` (reactive) |
| `ID` | ID type, typically `String` |
| `findByType(String type)` | Looks up an integrated tool by its type identifier |
| `findByKey(String key)` | Looks up a tool by its human-readable key (e.g., `"fleetmdm-server"`) with automatic tenant injection |

## Usage Example

```java
// Blocking implementation (e.g., for background jobs)
public interface IntegratedToolRepository
    extends BaseIntegratedToolRepository<Optional<IntegratedTool>, Boolean, String> {
    // inherits findByType and findByKey
}

// Reactive implementation (e.g., for API services)
public interface ReactiveIntegratedToolRepository
    extends BaseIntegratedToolRepository<Mono<IntegratedTool>, Mono<Boolean>, String> {
    // inherits findByType and findByKey
}

// Usage in a service
Mono<IntegratedTool> tool = reactiveRepo.findByKey("fleetmdm-server");
Optional<IntegratedTool> toolBlocking = blockingRepo.findByType("MDM");
```

## Notes

- `findByKey` integrates with `TenantAwareMongoTemplate`, which automatically injects the `tenantId` scoping queries to the correct tenant — no manual tenant filtering required in service code.
- The `B` type parameter is reserved for future boolean-returning operations (e.g., `existsByType`).
