MongoDB document entity representing a release version record stored in the `release_versions` collection, with optimistic locking and automatic audit timestamps.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `DEFAULT_ID` | `String` constant | Sentinel ID (`"default"`) for the primary/singleton version document |
| `id` | `String` | MongoDB document `_id` field |
| `documentVersion` | `Long` | `@Version` field enabling optimistic locking to prevent concurrent write conflicts |
| `version` | `String` | Indexed release version string (e.g., `"1.2.3"`) |
| `createdAt` | `Instant` | Auto-populated on document creation via `@CreatedDate` |
| `updatedAt` | `Instant` | Auto-updated on every save via `@LastModifiedDate` |

## Usage Example

```java
// Create and persist a new release version
ReleaseVersion releaseVersion = new ReleaseVersion();
releaseVersion.setId(ReleaseVersion.DEFAULT_ID);
releaseVersion.setVersion("2.1.0");

releaseVersionRepository.save(releaseVersion);

// Fetch the singleton default version document
Optional<ReleaseVersion> current = releaseVersionRepository
    .findById(ReleaseVersion.DEFAULT_ID);

current.ifPresent(rv -> {
    System.out.println("Version: " + rv.getVersion());
    System.out.println("Last updated: " + rv.getUpdatedAt());
});
```

## Notes

- Requires `@EnableMongoAuditing` on the Spring Boot application or a `MongoConfig` class for `@CreatedDate` / `@LastModifiedDate` to populate automatically.
- The `@Version` field (`documentVersion`) protects against lost updates — Spring Data MongoDB will throw `OptimisticLockingFailureException` on stale writes.
- The `DEFAULT_ID` constant is intended for a singleton pattern where a single canonical version document is maintained per deployment.
