MongoDB document entity representing a tenant-scoped ticket status definition, stored in the `ticket_statuses` collection with a compound unique index enforcing one entry per tenant per status kind.

## Key Components

| Member | Type | Description |
|---|---|---|
| `id` | `String` | MongoDB document identifier (`@Id`) |
| `tenantId` | `String` | Tenant isolation field (via `TenantScoped`) |
| `kind` | `TicketStatusKind` | Enum classifying the status (e.g., `AI_ASSISTANCE`, `TECH_REQUIRED`, `RESOLVED`, `ARCHIVED`) |
| `name` | `String` | Display label for the status |
| `color` | `String` | UI color value associated with the status |
| `position` | `String` | Ordering/sort position in the status list |
| `createdAt` / `updatedAt` | `Instant` | Auto-managed audit timestamps |

**Compound Index:** `tenant_kind_unique` enforces uniqueness on `(tenantId, kind)`, but only for the four built-in system kinds via a `partialFilter`. Custom statuses are excluded from this constraint.

## Usage Example

```java
TicketStatusDefinition status = TicketStatusDefinition.builder()
    .tenantId("tenant-abc")
    .kind(TicketStatusKind.RESOLVED)
    .name("Resolved")
    .color("#4ECDC4")
    .position("4")
    .build();

// Persist via Spring Data MongoDB repository
ticketStatusRepository.save(status);

// Lookup all statuses for a tenant
List<TicketStatusDefinition> statuses = ticketStatusRepository
    .findAllByTenantId("tenant-abc");
```

> The partial filter on the compound index means only system-defined `TicketStatusKind` values are uniqueness-constrained per tenant — allowing multiple custom statuses to coexist freely.
