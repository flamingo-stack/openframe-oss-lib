Fallback implementation of `ToolUpstreamResolver` that resolves upstream URLs for integrated tools by reading configuration from MongoDB via `ToolUrlService`. Used when no dedicated resolver is registered for a given tool.

## Key Components

| Member | Description |
|--------|-------------|
| `supportsToolId()` | Returns `Optional.empty()` to mark this as the catch-all fallback resolver |
| `resolveRest()` | Resolves the REST upstream URI using `ToolUrlType.API` |
| `resolveWs()` | Resolves the WebSocket upstream URI using `ToolUrlType.WS` |
| `resolveByType()` | Core logic — fetches a `ToolUrl` from `ToolUrlService` and delegates to `ProxyUrlResolver` to build the final `URI` |

## Usage Example

```java
// Automatically used by the gateway when no tool-specific resolver matches.
// Registration is handled via Spring component scan; no manual wiring needed.

// Internally, the gateway dispatcher resolves upstreams like:
URI upstream = defaultToolUpstreamResolver.resolveRest(integratedTool, request, "/api/tool");
// → reads ToolUrl from Mongo (type=API), builds proxied URI via ProxyUrlResolver

URI wsUpstream = defaultToolUpstreamResolver.resolveWs(integratedTool, request, "/ws/tool");
// → reads ToolUrl from Mongo (type=WS), builds proxied WebSocket URI
```

> **Note:** Throws `IllegalStateException` if the tool document has no URL configured for the requested `ToolUrlType`. Ensure every `IntegratedTool` document contains both `API` and `WS` entries if WebSocket proxying is expected.
