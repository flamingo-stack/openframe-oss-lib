Reactive REST controller that exposes tool integration endpoints under the `/tools` base path, handling health checks, connection tests, and transparent HTTP proxying for both direct API and agent-scoped requests.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `integrationService` | `IntegrationService` | Handles integration connection testing |
| `restProxyService` | `RestProxyService` | Forwards proxied HTTP requests to downstream tools |
| `healthCheck` | `GET /{toolId}/health` | Verifies connectivity for a registered tool |
| `testIntegration` | `POST /{toolId}/test` | Runs a connection test against a tool integration |
| `proxyApiRequest` | `ALL {toolId}/**` | Transparent reverse proxy for direct tool API calls |
| `proxyAgentRequest` | `ALL agent/{toolId}/**` | Transparent reverse proxy for agent-scoped tool calls |

## Usage Example

```java
// Health check a registered tool (e.g. via HTTP client)
// GET /tools/connectwise/health

// Test an integration connection
// POST /tools/connectwise/test

// Proxy an arbitrary API call to a tool
// GET /tools/connectwise/v4_6_release/apis/3.0/service/tickets

// Proxy an agent-scoped request
// POST /tools/agent/connectwise/v4_6_release/apis/3.0/service/tickets
```

## Behavior Notes

- All endpoints return `Mono<ResponseEntity<String>>` for non-blocking, reactive execution.
- Health and test endpoints gracefully return `400 Bad Request` with the error message on failure via `onErrorResume`.
- The proxy endpoints support all standard HTTP methods (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `OPTIONS`) and pass the raw request body and headers downstream via `RestProxyService`.
- `Authentication` is injected on health/test endpoints for security context availability but not explicitly used in routing logic.
