Reactive Spring service that enforces API rate limits across multiple time windows (minute, hour, day) using atomic Redis operations via `ReactiveRateLimitRepository`.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `isAllowed(String keyId)` | `Mono<Boolean>` | Checks all three time windows concurrently; returns `false` if any limit is exceeded |
| `getRateLimitStatus(String keyId)` | `Mono<RateLimitStatus>` | Returns current counters and exceeded flags for all windows without incrementing |
| `checkAndIncrement(...)` | `Mono<RateLimitResult>` | Atomically checks and increments the counter for a given window via repository |
| `getStatus(...)` | `Mono<RateLimitResult>` | Reads current count for a window without side effects |
| `createFailureResult(long limit)` | `RateLimitResult` | Builds a safe fallback result on Redis error, respecting `failOpen` flag |
| `createEmptyStatus(String keyId)` | `RateLimitStatus` | Builds a zeroed-out status on error for `getRateLimitStatus` fallback |

**Configuration properties (injected):**

- `openframe.rate-limit.redis-ttl` — Redis key TTL in seconds
- `openframe.rate-limit.fail-open` — When `true`, allows requests through on Redis failure

## Usage Example

```java
// Check if a request is permitted before processing
rateLimitService.isAllowed("api-key-abc123")
    .flatMap(allowed -> {
        if (!allowed) {
            return Mono.error(new RateLimitExceededException("Too many requests"));
        }
        return proceedWithRequest();
    });

// Fetch status for monitoring/debugging
rateLimitService.getRateLimitStatus("api-key-abc123")
    .doOnNext(status -> {
        log.info("Day usage: {}/{}", status.getDayRequests(), status.getDayLimit());
    })
    .subscribe();
```

> All three windows (`MINUTE`, `HOUR`, `DAY`) are evaluated concurrently via `Mono.zip`, minimising latency. Redis errors fall back to the `failOpen` policy rather than blocking traffic.
