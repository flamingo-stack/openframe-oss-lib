A decorator for Spring's `WebSocketService` that enforces JWT-based session lifecycle management on secured WebSocket endpoints in the OpenFrame gateway.

## Key Components

| Member | Description |
|--------|-------------|
| `CLOCK_SKEW_SECONDS` | 60-second buffer added to JWT expiry to align with Spring Security's default clock skew tolerance |
| `sessionRegistry` | Thread-safe `ConcurrentHashMap` tracking active sessions with creation time, path, and subject |
| `SessionInfo` | Record holding per-session metadata (`createdAt`, `path`, `sub`) |
| `handleRequest()` | Intercepts WebSocket upgrade requests; applies JWT enforcement only on secured endpoints |
| `isSecuredEndpoint()` | Checks if the request path matches any known secured endpoint prefixes (tools API/agent, NATS WS paths) |
| `scheduleSessionRemoveJob()` | Schedules a reactive `Mono.delay` to force-close the session when the JWT expires |
| `processSessionClosedEvent()` | Subscribes to the session close status to clean up the registry, dispose the expiry job, and emit traffic metrics |
| `formatLifetime()` | Formats session duration in human-readable `min`/`sec` form for logging |

## Usage Example

The decorator wraps the default `WebSocketService` bean and is wired via `WebSocketGatewayConfig`:

```java
@Bean
public WebSocketService webSocketService(
    WebSocketService defaultWebSocketService,
    RequestJwtClaimsReader jwtReader,
    GatewayTrafficMetrics metrics,
    WebSocketLoggingProperties loggingProps
) {
    return new WebSocketServiceSecurityDecorator(
        defaultWebSocketService,
        jwtReader,
        metrics,
        loggingProps
    );
}
```

At runtime, on a secured path (e.g. `/tools/api/ws`), the decorator:

1. Opens the session and registers it in `sessionRegistry`
2. Reads the JWT expiration and schedules an auto-close after `expiresAt + 60s`
3. Observes `session.closeStatus()` — if the client disconnects first, the scheduled job is disposed
4. Emits `webSocketOpened` / `webSocketClosed` metrics for traffic observability

> **Note:** Secured endpoint paths are currently hardcoded. A TODO exists to rely on Spring Security configuration instead of explicit path matching.
