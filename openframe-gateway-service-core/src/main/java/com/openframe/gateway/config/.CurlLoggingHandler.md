A Netty `ChannelDuplexHandler` that intercepts outbound HTTP requests and reconstructs them as equivalent `curl` commands for debug logging.

## Key Components

| Member | Description |
|--------|-------------|
| `curl` | `StringBuilder` that accumulates the curl command string across handler invocations |
| `isRequest` | Flag tracking whether the current write sequence belongs to an HTTP request |
| `TARGET_URI` | `AttributeKey<URI>` used to retrieve the resolved target URI from channel attributes (key: `"target_uri"`) |
| `channelRegistered()` | Logs a debug message when the channel is registered |
| `write()` | Core interception logic — builds the curl command from `HttpRequest` headers/method and `HttpContent` body, then logs it on `LastHttpContent` |

## Behavior

```text
HttpRequest  → captures method, full URL, and headers (Host excluded)
HttpContent  → appends --data-raw body if readable
LastHttpContent → flushes the complete curl string to debug log, resets state
```

## Usage Example

Register the handler in a Netty pipeline (e.g., inside a Spring Cloud Gateway `NettyRoutingFilter` or a custom `NettyConfiguration`):

```java
@Bean
public NettyServerCustomizer nettyServerCustomizer() {
    return httpServer -> httpServer.doOnChannelInit((observer, channel, remoteAddress) -> {
        channel.pipeline().addLast(new CurlLoggingHandler());
    });
}
```

Set the target URI as a channel attribute before the request is written:

```java
channel.attr(AttributeKey.<URI>valueOf("target_uri")).set(targetUri);
```

Enable debug output in `application.yml`:

```yaml
logging:
  level:
    com.openframe.gateway.config.CurlLoggingHandler: DEBUG
```

> **Note:** This handler is intended for development/debugging only. Avoid enabling it in production environments as it logs full request bodies and headers, which may expose sensitive data.
