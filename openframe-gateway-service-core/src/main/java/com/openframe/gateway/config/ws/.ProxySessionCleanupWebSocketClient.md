A decorator for Spring's `WebSocketClient` that ensures downstream proxy WebSocket sessions are properly closed after relay completion, with optional debug logging per path.

## Key Components

| Element | Description |
|---|---|
| `ProxySessionCleanupWebSocketClient` | Implements `WebSocketClient`; wraps a delegate client to add cleanup logic |
| `delegate` | The underlying `WebSocketClient` that performs the actual connection |
| `loggingProperties` | `WebSocketLoggingProperties` used to enable per-path debug logging |
| `wrapHandler(URI, WebSocketHandler)` | Decorates the original handler with `doFinally` cleanup logic |
| `CLOSE_TIMEOUT` | 5-second timeout applied when force-closing a lingering session |

## Behavior

After the relay `Mono<Void>` completes (any terminal signal), the wrapper checks whether the proxy session is still open. If so, it issues a `CloseStatus.GOING_AWAY` close with a 5-second timeout, logs the outcome, and silently swallows errors to avoid propagating close failures upstream.

## Usage Example

```java
@Bean
public WebSocketClient webSocketClient(
        ReactorNettyWebSocketClient nettyClient,
        WebSocketLoggingProperties loggingProperties) {

    return new ProxySessionCleanupWebSocketClient(nettyClient, loggingProperties);
}

// The client is then used transparently like any other WebSocketClient:
webSocketClient.execute(
    URI.create("ws://backend-service/ws/chat"),
    headers,
    session -> session.receive()
        .then()
).subscribe();
```

> **Note:** This client is typically registered as a Spring bean and injected into the gateway's WebSocket proxy routing configuration. It adds no behavioral changes to the happy path — cleanup only triggers when a session remains open after relay termination.
