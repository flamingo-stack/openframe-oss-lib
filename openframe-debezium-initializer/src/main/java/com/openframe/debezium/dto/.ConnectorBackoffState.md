Tracks the backoff state for a Debezium connector, managing consecutive failure counts and restart eligibility based on calculated delay windows.

## Key Components

| Member | Type | Description |
|--------|------|-------------|
| `consecutiveFailures` | `int` | Running count of sequential connector failures |
| `nextEligibleRestart` | `Instant` | Timestamp after which a restart is permitted |
| `recordFailure(long backoffMs)` | `void` | Increments failure count and schedules the next eligible restart |
| `isEligibleForRestart()` | `boolean` | Returns `true` if the current time has passed `nextEligibleRestart` |
| `getConsecutiveFailures()` | `int` | Returns the current failure count |
| `getNextEligibleRestart()` | `Instant` | Returns the scheduled restart eligibility timestamp |

## Usage Example

```java
ConnectorBackoffState state = new ConnectorBackoffState();

// Simulate a failure with a 5-second backoff
state.recordFailure(5000L);

System.out.println(state.getConsecutiveFailures());   // 1
System.out.println(state.isEligibleForRestart());     // false (within backoff window)

// After backoff period elapses...
if (state.isEligibleForRestart()) {
    restartConnector();
}
```

## Behavior Notes

- **Initial state**: `consecutiveFailures = 0`, `nextEligibleRestart = Instant.MIN` — connector is immediately eligible for restart on creation.
- **Backoff accumulation**: Each call to `recordFailure` pushes `nextEligibleRestart` forward by `backoffMs` milliseconds from the moment of failure — callers are responsible for applying exponential or fixed backoff strategies when computing `backoffMs`.
- **No reset logic**: Failure count is never decremented internally; consumers must instantiate a new `ConnectorBackoffState` or manage resets externally upon successful recovery.
