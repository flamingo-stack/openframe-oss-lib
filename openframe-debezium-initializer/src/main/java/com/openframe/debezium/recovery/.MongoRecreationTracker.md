MongoDB-backed implementation of `RecreationTracker` that enforces a rolling one-hour rate limit on Debezium connector recreations across all management replicas.

## Key Components

| Member | Description |
|---|---|
| `WINDOW` | Fixed 1-hour rolling window for rate limiting |
| `maxPerHour` | Configurable max recreations per hour (default: `1`) via `openframe.debezium.recovery.max-recreations-per-hour` |
| `canRecreate(String baseName)` | Counts recent recreation events for the connector; returns `false` if the limit is reached or MongoDB is unreachable (fail-closed) |
| `record(String baseName)` | Persists a new `ConnectorRecreationEvent` and purges stale events outside the rolling window |
| `cutoff()` | Computes the rolling window boundary (`now - 1 hour`) |

## Behavior Notes

- **Distributed-safe**: state is stored in MongoDB, so limits are shared across all replicas
- **Fail-closed**: if MongoDB is unreachable during `canRecreate`, recreation is denied to prevent runaway connector loops
- **Self-purging**: `record()` deletes expired events on each write, eliminating the need for a TTL index

## Usage Example

```java
// Activated only when property is true
// openframe.debezium.recovery.recreation.enabled=true

@Autowired
private RecreationTracker recreationTracker;

String baseName = "orders-connector";

if (recreationTracker.canRecreate(baseName)) {
    recreationTracker.record(baseName);
    // proceed with connector recreation
} else {
    log.warn("Skipping recreation for '{}' — rate limit exceeded", baseName);
}
```

## Configuration

```yaml
openframe:
  debezium:
    recovery:
      recreation:
        enabled: true
      max-recreations-per-hour: 1  # default
```
